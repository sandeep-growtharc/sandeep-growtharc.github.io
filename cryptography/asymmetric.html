Section 1

Core concepts — the key pair

Asymmetric cryptography uses two mathematically linked keys that do opposite jobs. What one encrypts, only the other can decrypt. They are always generated together as a pair.

🔓

Public key

Shared freely with everyone. Used to encrypt data sent to you, and to verify signatures you create. Lives inside your SSL certificate.

🔒

Private key

Never leaves your server. Used to decrypt data encrypted with your public key, and to create digital signatures others can verify.

The analogy: think of a padlock and key. You hand out open padlocks (your public key) to anyone who wants to send you something. They click it shut (encrypt). Only your physical key (private key) can open it. You never hand out your key — just the padlock.

Section 2

How asymmetric crypto works

The same key pair is used in two opposite ways depending on the goal.

For confidentiality — encryption

Alice wants to send Bob a secret message. She encrypts it with Bob's public key. Only Bob (private key holder) can decrypt it.

Encryption flow

For authenticity — digital signatures

Bob signs a document with his private key. Anyone with Bob's public key can verify it was really him and the document wasn't tampered with.

Digital signature flow

Encryption rule

Encrypt with public key → only private key can decrypt.

Signature rule

Sign with private key → anyone with the public key can verify.

Section 3

What is an SSL certificate?

A public key by itself is anonymous — anyone could claim it belongs to google.com. A certificate is a signed document that binds a public key to a verified identity.

X.509 certificate anatomy

The CA's signature at the bottom is what makes a certificate trustworthy. It says: "I, Let's Encrypt, verified that this public key genuinely belongs to example.com."

Section 4

Certificate Authority & chain of trust

Your browser comes pre-loaded with a list of trusted Root CAs. It verifies certificates by walking up the chain until it reaches one it recognises.

🏛️ Root CA

e.g. DigiCert, ISRG (Let's Encrypt), Comodo
Pre-trusted by your OS & browser. Kept offline (air-gapped).

↓ signs

🏢 Intermediate CA

Does the day-to-day signing. Protects Root CA from exposure.
If compromised, only this branch is revoked.

↓ signs

📄 Your website certificate

example.com — contains your public key + domain identity.
This is what the browser receives in the TLS handshake.

Break any link in the chain (expired cert, wrong CA, revoked intermediate) and the browser shows a ⚠️ warning and blocks the connection.
Section 5

Public keys in TLS/SSL

The public key has two jobs during a TLS connection: authenticate the server's identity, and bootstrap a shared secret for the session.

The TLS handshake step by step

1
ClientHelloBrowser sends: supported cipher suites, TLS version, random nonce.
2
ServerHello + CertificateServer responds with its SSL certificate — containing its public key.
3
Certificate verificationBrowser walks the chain of trust up to a trusted Root CA. Checks domain match and expiry.
4
Key exchange — public key used hereBrowser generates a pre-master secret and encrypts it with the server's public key. Only the server can decrypt it.
5
Server decrypts with private keyServer decrypts the pre-master secret. Both sides derive the same symmetric session key.
6
Encrypted channel (HTTPS)All further traffic uses AES (symmetric). Public key's job is done.
TLS 1.3 note: RSA key exchange was removed. ECDHE (Diffie-Hellman) is used instead, providing forward secrecy — past sessions can't be decrypted even if the private key is stolen later. The public key is still used for authentication/signature verification.

Why switch to AES after handshake?

RSA is 100–1000× slower than AES. Public key crypto is used once to bootstrap trust. AES encrypts the actual data stream.

What is forward secrecy?

Ephemeral (one-time) key pairs are generated per session. Old sessions can't be decrypted later even with the private key.

Hands-on Lab

OpenSSL command-line labs

All you need is openssl — pre-installed on macOS and most Linux distros. Check with openssl version.

Lab 1

Generate & inspect a key pair

Goal — generate an RSA key pair and understand what the numbers inside actually mean.
Generate a 2048-bit RSA private key
openssl genrsa -out private.pem 2048
Extract the public key from it
openssl rsa -in private.pem -pubout -out public.pem
Decode the private key — see the actual numbers
openssl rsa -in private.pem -text -noout
FieldWhat it is
modulusN = p × q (two secret primes multiplied). This is public.
publicExponentAlways 65537 (0x10001). This is public.
privateExponentThe secret d, derived from p and q. Keep secret.
prime1, prime2The two secret primes p and q. The core secret.
Decode just the public key — notice what's missing
openssl rsa -pubin -in public.pem -text -noout

Only modulus and publicExponent are visible. The security of RSA relies on the impossibility of factoring N back into p and q for 2048-bit keys.

Lab 2

Encrypt & decrypt a message

Goal — encrypt with the public key, decrypt with the private key. Proves only the private key holder can read it.
Create a test message
echo "Hello from the TLS lab!" > message.txt
Encrypt with the PUBLIC key (anyone can do this)
openssl rsautl -encrypt \
  -inkey public.pem \
  -pubin \
  -in message.txt \
  -out message.enc
Decrypt with the PRIVATE key (only key holder can do this)
openssl rsautl -decrypt \
  -inkey private.pem \
  -in message.enc \
  -out decrypted.txt

cat decrypted.txt
You should see: Hello from the TLS lab! — exactly what a TLS handshake does when bootstrapping a session key.
Lab 3

Sign & verify a message

Goal — sign with the private key, verify with the public key. Then intentionally tamper with the message to see verification fail.
Sign with the PRIVATE key
openssl dgst -sha256 \
  -sign private.pem \
  -out message.sig \
  message.txt
Verify with the PUBLIC key
openssl dgst -sha256 \
  -verify public.pem \
  -signature message.sig \
  message.txt
Expected output: Verified OK
Tamper with the message and verify again
echo "Tampered!" > message.txt

openssl dgst -sha256 \
  -verify public.pem \
  -signature message.sig \
  message.txt
Expected output: Verification Failure — any change to the data invalidates the signature. This is how TLS detects man-in-the-middle tampering.
Lab 4

Create a self-signed certificate

Goal — bundle your public key with identity info and sign it yourself. Useful for internal servers and testing.
Generate private key + self-signed cert in one command
openssl req -x509 -new \
  -key private.pem \
  -sha256 \
  -days 365 \
  -out cert.pem \
  -subj "/CN=myserver.local/O=My Lab/C=US"
Inspect the certificate in human-readable form
openssl x509 -in cert.pem -text -noout

Notice: the Issuer and Subject are the same — that's the tell for a self-signed cert. Browsers show a warning because no trusted CA signed it.

Extract just the public key from the cert
openssl x509 -in cert.pem -pubkey -noout

This should match your public.pem — the certificate is just a wrapper tying the public key to an identity.

Lab 5

Inspect a real production certificate

Goal — connect to a live HTTPS server and read its certificate. See everything above in a real-world context.
Fetch and decode GitHub's certificate
openssl s_client -connect github.com:443 \
  -showcerts </dev/null 2>/dev/null \
  | openssl x509 -text -noout
View the full certificate chain (CA hierarchy)
openssl s_client -connect github.com:443 \
  -showcerts </dev/null 2>/dev/null \
  | grep "subject\|issuer"

You'll see 2–3 certificates: your server cert → Intermediate CA → Root CA. This is the chain of trust rendered live.

Check certificate expiry (useful in scripts/monitoring)
openssl s_client -connect github.com:443 \
  </dev/null 2>/dev/null \
  | openssl x509 -noout -dates
Section 6

Creating a CSR for a real CA

A CSR (Certificate Signing Request) is the file you send to a CA asking them to sign your public key and vouch for your identity. Your private key never leaves your server.

CSR flow — your server to CA and back
Generate a new private key + CSR together
openssl req -new \
  -newkey rsa:2048 \
  -keyout private.pem \
  -out server.csr \
  -subj "/CN=yourdomain.com/O=Your Org/OU=Engineering/C=US/ST=California/L=San Jose"
FieldMeaning
CN=Common Name — the primary domain. Must exactly match what users browse to.
O=Organisation name
OU=Organisational unit (team/department)
C=Country (2-letter ISO code, e.g. US, GB, IN)
ST=State or Province
L=City / Locality
Verify your CSR looks correct before sending
openssl req -in server.csr -text -noout
openssl req -in server.csr -verify -noout
Section 7

Multi-domain certificates — Subject Alternative Names

Browsers stopped respecting the CN field for hostname matching in 2017. All modern certificates require Subject Alternative Names (SANs) to list every domain covered.

A certificate without SANs will fail hostname verification in all modern browsers, even if the CN looks correct.
Create a SAN config file
cat > san.cnf << 'EOF'
[req]
default_bits       = 2048
prompt             = no
default_md         = sha256
distinguished_name = dn
req_extensions     = req_ext

[dn]
CN = yourdomain.com
O  = Your Org
C  = US

[req_ext]
subjectAltName = @alt_names

[alt_names]
DNS.1 = yourdomain.com
DNS.2 = www.yourdomain.com
DNS.3 = api.yourdomain.com
DNS.4 = staging.yourdomain.com
IP.1  = 192.168.1.10
EOF
Generate CSR using the config
openssl req -new \
  -key private.pem \
  -out server_san.csr \
  -config san.cnf
Verify the SANs are embedded in the CSR
openssl req -in server_san.csr -text -noout \
  | grep -A4 "Subject Alternative"
Wildcard certs: use DNS.1 = *.yourdomain.com to cover all subdomains with one cert. Wildcards only go one level deep — *.example.com does NOT cover sub.sub.example.com.
Section 8

Simulating a CA — the full PKI flow

This exercise makes you the CA. It's invaluable for understanding internal PKI (corporate VPNs, internal tools, mutual TLS between microservices).

Full PKI

Create a CA, sign a CSR, verify the chain

Step A — create your CA key pair (you are now the CA)
openssl genrsa -out ca-private.pem 4096

openssl req -x509 -new \
  -key ca-private.pem \
  -sha256 -days 1825 \
  -out ca-cert.pem \
  -subj "/CN=My Lab CA/O=Lab/C=US"
Guard ca-private.pem with extreme care. Anyone who holds it can issue certificates trusted by any system that trusts your CA. In real PKIs the Root CA private key is stored offline on an HSM.
Step B — CA signs the server CSR
openssl x509 -req \
  -in server.csr \
  -CA ca-cert.pem \
  -CAkey ca-private.pem \
  -CAcreateserial \
  -out server-cert.pem \
  -days 365 \
  -sha256
Step C — verify the cert chains to your CA
openssl verify -CAfile ca-cert.pem server-cert.pem
Expected: server-cert.pem: OK — this is exactly what a browser does during a TLS handshake.
Step D — inspect the final signed cert
openssl x509 -in server-cert.pem -text -noout

The Issuer field now reads "My Lab CA" — proving it was signed by your CA. The Subject is your server. The Signature at the bottom is the CA's cryptographic stamp.

Section 9

Let's Encrypt with Certbot

For real, browser-trusted certificates. Free, automated, and the most widely used CA on the internet. Certbot handles the CSR creation, domain verification, and certificate retrieval automatically.

Install Certbot — Ubuntu/Debian
sudo apt install certbot
Install Certbot — macOS
brew install certbot
Option A — standalone mode (port 80 must be free)
sudo certbot certonly --standalone \
  -d yourdomain.com \
  -d www.yourdomain.com
Option B — DNS challenge (required for wildcard certs)
sudo certbot certonly --manual \
  --preferred-challenges dns \
  -d "*.yourdomain.com" \
  -d yourdomain.com
Where your cert files land
/etc/letsencrypt/live/yourdomain.com/
  ├── privkey.pem    ← your private key (guard this)
  ├── cert.pem       ← your certificate
  ├── chain.pem      ← intermediate CA cert
  └── fullchain.pem  ← cert + chain combined (use this in nginx/apache)
Auto-renew test (certs expire after 90 days)
sudo certbot renew --dry-run
Expired certs are one of the most common HTTPS outage causes. Always confirm the auto-renew cron/systemd timer is active in production.
nginx config snippet
ssl_certificate     /etc/letsencrypt/live/yourdomain.com/fullchain.pem;
ssl_certificate_key /etc/letsencrypt/live/yourdomain.com/privkey.pem;
Quick Reference

OpenSSL cheatsheet

Key generation

# RSA private key
openssl genrsa -out private.pem 2048

# Extract public key
openssl rsa -in private.pem -pubout -out public.pem

# Inspect private key
openssl rsa -in private.pem -text -noout

# Inspect public key
openssl rsa -pubin -in public.pem -text -noout

Encryption & decryption

# Encrypt with public key
openssl rsautl -encrypt -inkey public.pem -pubin -in file.txt -out file.enc

# Decrypt with private key
openssl rsautl -decrypt -inkey private.pem -in file.enc -out file.txt

Signing & verification

# Sign
openssl dgst -sha256 -sign private.pem -out file.sig file.txt

# Verify
openssl dgst -sha256 -verify public.pem -signature file.sig file.txt

Certificates

# Self-signed cert
openssl req -x509 -new -key private.pem -sha256 -days 365 \
  -out cert.pem -subj "/CN=example.com/O=Org/C=US"

# Inspect a cert
openssl x509 -in cert.pem -text -noout

# Check expiry
openssl x509 -in cert.pem -noout -dates

# Extract public key from cert
openssl x509 -in cert.pem -pubkey -noout

CSR

# Generate key + CSR
openssl req -new -newkey rsa:2048 -keyout private.pem \
  -out server.csr -subj "/CN=yourdomain.com/O=Org/C=US"

# Inspect CSR
openssl req -in server.csr -text -noout

# Verify CSR signature
openssl req -in server.csr -verify -noout

# CA signs a CSR
openssl x509 -req -in server.csr -CA ca-cert.pem \
  -CAkey ca-private.pem -CAcreateserial \
  -out server-cert.pem -days 365 -sha256

# Verify cert against CA
openssl verify -CAfile ca-cert.pem server-cert.pem

Live server inspection

# Fetch and decode a live cert
openssl s_client -connect github.com:443 \
  -showcerts </dev/null 2>/dev/null | openssl x509 -text -noout

# View cert chain
openssl s_client -connect github.com:443 \
  -showcerts </dev/null 2>/dev/null | grep "subject\|issuer"

# Check expiry of live cert
openssl s_client -connect github.com:443 \
  </dev/null 2>/dev/null | openssl x509 -noout -dates
